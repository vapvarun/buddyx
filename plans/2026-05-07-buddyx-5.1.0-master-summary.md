## 4. Architecture pillars (what to port to buddyx-pro)

### 4.1 Customizer_Framework (replaces Kirki)

Lives at `inc/Customizer_Framework/` — 19 PHP files, drop-in Kirki-shape API.

Drop-in compat with:
- `\Kirki\Field\X::add(...)` → `\BuddyX\Buddyx\Customizer_Framework\Field::add('type', $args)`
- All 12 custom controls (typography, color-alpha, multiselect-checkbox, etc.)
- Active_Callback compiles Kirki-shape arrays into closures (supports `==`, `!=`, `>=`, `<=`, `in`, `contains` operators)
- Sanitize callbacks honor Kirki's loose-equality behavior (`'on' == '1'`)

**Porting to buddyx-pro:** copy the entire `inc/Customizer_Framework/` directory verbatim. Update namespace from `BuddyX\Buddyx` to `BuddyX\Buddyx_Pro`. The framework was designed as portable from day one for exactly this purpose.

### 4.2 Tokens system

Lives at `inc/Tokens/Component.php` (one file, ~600 LOC).

Responsibilities:
- Reads customizer color settings → emits `:root { --bx-color-* : ... }`
- Maintains parallel legacy aliases (`--color-*`, `--button-*`, `--global-*`) for backward compat with 5.0.x stylesheets
- Auto-derives 9 variants per customer-driven base color (light, dark, hover, focus, etc.)
- Reconciles customizer values with style-variation overlays (customer wins)
- Emits dark-mode token set under `[data-bx-mode="dark"] :root`
- FOUC-prevention head script applies stored mode before render

96-token canonical taxonomy lives at `assets/css/src/_bx-tokens.css`.

**Porting to buddyx-pro:** copy `inc/Tokens/Component.php` and `assets/css/src/_bx-tokens.css`. Update the 36-customizer-field map (`$simple_color_tokens` array) if buddyx-pro adds Pro-specific colors. Otherwise it's verbatim.

### 4.3 Color_Mode_Toggle

Lives at `inc/Color_Mode_Toggle/Component.php` — visitor-side dark-mode button.

Responsibilities:
- Reads `site_color_mode_toggle_show` + `site_color_mode_toggle_position` customizer settings
- Renders sun/moon/monitor SVG button in header + mobile menu
- JS at `assets/js/src/color-mode-toggle.js` cycles state, writes localStorage, syncs `<html data-bx-mode>`
- Selectively-enqueued JS (zero asset cost when toggle disabled)

**Porting to buddyx-pro:** copy verbatim. Same UX pattern works for Pro.

### 4.4 Starter_Content

Lives at `inc/Starter_Content/Component.php` — fresh-install demo via WP starter content API.

Responsibilities:
- Registers `add_theme_support('starter-content', $config)` on `after_setup_theme` priority 12
- 7 demo pages composed from existing patterns
- Sidebar widgets, primary nav menu, theme_mods seeding `site_color_mode` and toggle visibility

**Porting to buddyx-pro:** Pro likely ships richer demo (premium patterns, Pro-only sections). Use the same component pattern but expand `posts()` array. Ensure no Pro plugin dependencies in starter content (must stay WP.org review-clean for the free counterpart).

### 4.5 Speculation_Rules

Lives at `inc/Speculation_Rules/Component.php` — WP 6.4+ Speculation Rules API for predictive prefetch.

Already shipped in master, no 5.1.0 changes needed for buddyx-pro.

## 7. Lessons learned (apply to buddyx-pro)

These are non-obvious gotchas the 5.1.0 work surfaced. Skip them on buddyx-pro and you'll repeat the bug.

1. **`.post` and `.page` are body classes too.** Targeting them as article-card selectors hijacks `body_background_color` on every static-page template. Use `.hentry` (universal on articles, never on body).

2. **`.button` is too greedy.** WP and BuddyPress use `.button` for many low-emphasis links (notification footer, BP friend buttons). Don't add it to the primary button rule — per-context stylesheets already scope it correctly.

3. **Customizer Framework's `output` config emits its default value as inline CSS even when `active_callback` is false.** When migrating fields, leave `default` empty for any field where you want the active_callback gate to actually gate. (This was the sub-header background bug.)

4. **`wc_get_cart_url()` falls back to `home_url()` when the cart page is deleted.** Always guard the cart icon render with: WC active + valid cart page + cart URL ≠ home URL.

5. **`fresh_site` flips to 0 on first published post.** Customers updating from older versions never see starter content — that's by design and you don't need to add migration paths.

6. **Pre-commit hook auto-runs eslint + stylelint.** Add JSDoc `@param` types to every function param; otherwise commit fails. Don't `--no-verify`.

7. **Phase 4 surface-BG rules cause empty colored regions when content is short.** Body should be the only token-driven surface; main, sidebar, article cards inherit. Phase 4 originally had layered cards which broke on directory + WC pages.

8. **Token defaults must match master legacy hex when there's no customizer-driven source.** `presence-online` was `#16a34a` in the new taxonomy but `#4caf50` in master — direct visible drift on customer sites. Preferred: change the default, not the rendered value.

9. **`getComputedStyle().backgroundColor` returns `rgba(0,0,0,0)` for transparent BG.** Don't trust naive rgb→hex converters; check the alpha channel first or you'll diagnose ghost bugs that don't exist.

10. **Build pipeline reads from `assets/css/src/`, NOT `assets/css/`.** Editing `assets/css/global.min.css` directly will be overwritten on next build. Always edit src.

## 8. buddyx-pro Kirki removal sequence (recommended)

Mirror the 5.1.0 commit order. Each phase has acceptance gates from the corresponding 5.1.0 plan doc.

1. **Customizer_Framework port** — copy `inc/Customizer_Framework/`, update namespace, run customizer-options-audit Pass 1 + 2 (byte-identical theme_mods round-trip)
2. **Customizer field migration** — replace each `\Kirki\Field\X::add()` with `\BuddyX\Buddyx_Pro\Customizer_Framework\Field::add('type', $args)`. Use the audit doc's Pass 3 (real customer DB) before tagging.
3. **Tokens system** — copy `inc/Tokens/Component.php` + `_bx-tokens.css`. Map Pro-specific customizer fields if they exist beyond the 36 in BuddyX free.
4. **Phase 4 stylesheet cleanup** — apply per-stylesheet token migrations following the 4a/b/c/d sub-phases. Apply lessons #7 (no surface-BG, no .post/.page) from the start.
5. **Color_Mode_Toggle port** — copy verbatim, no Pro-specific changes needed.
6. **Starter_Content port** — extend with Pro-only patterns/pages.
7. **Customer parity stress test** — 73/73 field check before tagging.
8. **Pass 3 staging regression sweep** — same blocker as BuddyX free; needs real customer DB on staging.
